#### 1.20.0 - 2026-06-23

- Document Sync: removed the two parameters that were already marked **legacy** — **Lease Token (Legacy)** (Download Document File / Report Transfer Status) and **SharePoint URL (Legacy)** (Report Transfer Status). The node no longer sends `leaseToken`/`token` to the service and no longer emits `leaseToken` on its output items; the service ignores these values anyway (claim locking is in-memory per tenant). For the transfer payload use the **Payload Fields** (e.g. key `sharePointUrl`). The unrelated **SharePoint URL** field on **Create Document** (`createDokumentSharePointUrl`) is unchanged. **Migration:** workflows that still had a value in "Lease Token (Legacy)" or "SharePoint URL (Legacy)" can drop it; if you relied on the SharePoint URL shortcut in Report Transfer Status, move it into a Payload Field with key `sharePointUrl`.

#### 1.19.0 - 2026-06-23

- Document Sync / Report SharePoint Transfer Status: new **Payload Fields** (key/value) area to build a custom flat JSON payload for the entry (e.g. `sharePointUrl`). Keys must be non-empty and unique (duplicates/empties raise an error), values are strings, total payload limited to 8 KB. The legacy single **SharePoint URL** field still works as a fallback when no Payload Fields are set. **Additive / non-breaking.**
- Document Sync / Check New Documents: DELETE events now return the **full stored `payload`** of the correlated prior upload (or `null` after retention), alongside the existing derived `sharePointUrl`. CREATE events for mail documents (`Typ=2`) now include an **`email`** object (`from`, `to`, `subject`, `date`).
- Document Sync / Create Document: the **SharePoint URL** is now **optional** — create a document without a SharePoint reference (e.g. when the storage export is handled later by WF1 via "Also Export to Storage").
- Service: `/api/sharepoint-transfer-status` accepts a generic `payload` object (stored flat) in addition to the legacy `sharePointUrl`; `/api/create-new-dokument` no longer requires `sharePointUrl`. No DB schema change.

#### 1.18.0 - 2026-06-23

- Document Sync / Download Document File & Report SharePoint Transfer Status: the **Lease Token** parameter is now **optional and marked legacy** (renamed display to "Lease Token (Legacy)"). The service no longer validates `leaseToken`/`token` — claim locking moved to an in-memory, per-tenant lock — so the field can be left empty. **Non-breaking:** the node still sends the value if provided, and the service still returns a dummy `leaseToken` in the Check New Documents response for backward compatibility. Full removal of `leaseToken`/`token` from request, response and `downloadUrl` is planned for a future **major** version (Phase 2).

#### 1.17.0 - 2026-06-22

- Document Sync / Create Document: new **Also Export to Storage (Disable Loop Guard)** option (off by default). When off, an imported document is created in TaxMetall but is not re-exported (the ERP sync trigger skips its own echo). When on, the created document is additionally queued for export (WF1) — useful to mirror imported mails/documents to your external storage (e.g. SharePoint) as well. Sent as `suppressLoopGuard` to `/api/create-new-dokument`; the service writes a distinct source marker so the trigger does not suppress the event. No infinite loop: the export path does not create another TaxMetall document.

#### 1.16.0 - 2026-06-22

- Document Sync / Report SharePoint Transfer Status: the failure text is now sent to the service under `errorMessage` (previously `error`, which the service ignored — the custom message was silently dropped and only a generic default was logged). The service accepts the old `error` key as a fallback, so older node versions keep working. The recorded log entry now contains the message you provide.
- Document Sync / Download Document File: file names are now parsed identically to "Check & Download New Documents", including RFC 5987 percent-decoding of the `Content-Disposition` `filename*` value — so downloads with non-ASCII names (umlauts, etc.) get the correct file name instead of a percent-encoded one. The shared parser also decodes defensively (a malformed value falls back to the raw name instead of throwing).

#### 1.15.0 - 2026-06-18

- **BREAKING:** Renamed the internal "WF2" naming to "create-new-dokument" throughout. The Document Sync / Create Document parameters were renamed from `wf2…` to `createDokument…` (`createDokumentBereich`, `createDokumentBelegnummer`, `createDokumentSharePointUrl`, `createDokumentEmailAn`, `createDokumentEmailFields`, `createDokumentAttachAllBinaries`, `createDokumentAttachments`, `createDokumentAllowDuplicates`). **Existing workflows that use "Create Document" must re-enter these fields.** No API change — the request payload to `/api/create-new-dokument` is identical; only the n8n parameter identifiers changed.

#### 1.14.3 - 2026-06-18

- Document Sync / Report SharePoint Transfer Status: the "Success" parameter is now a text field instead of a checkbox, so the value can be passed through from a previous step via an expression (e.g. the upstream node's success indicator) rather than being toggled manually. Accepted truthy values are `true`, `1`, `yes`, `success` (case-insensitive); anything else is treated as a failed transfer. The node coerces the value to a real boolean before calling the service, so the API request is unchanged. Because the value can now be an expression, the "SharePoint URL" and "Error Message" fields are always shown for this operation (n8n cannot evaluate an expression to drive conditional visibility) — fill whichever applies. Removed stale "retry / dead after maximum" wording from the field descriptions (failed transfers are retried unconditionally on the next poll).

#### 1.14.2 - 2026-06-18

- Document Sync: updated the node to the service's new event contract. Each claimed document now exposes `eventTyp` (`1`/`2`/`3`) and `aktion` (`CREATE`/`EDIT`/`DELETE`) instead of the former `eventType` (`document.created`/`document.deleted`). "Check & Download New Documents" now decides whether to fetch a file by the presence of `downloadUrl` (set for CREATE/EDIT only), so downloads work again against the current service — previously the renamed field left `hasFile` always false and no file was downloaded. README aligned with the new contract: `check-new-documents` can return `busy: true` (another batch in progress, nothing claimed), delete correlation is by file path, and a failed transfer keeps its work-queue row and is retried unconditionally on the next poll (no retry counter, no `dead` state).

#### 1.14.0 - 2026-06-16

- Document Sync / Check & Download New Documents: the operation now always returns something. When no documents are found it emits a single summary item `{ success: false, count: 0, description: "No new documents found" }` instead of empty output, so you can drive a "nothing to do" branch. When documents are found, every item additionally carries `success: true`, `count` (total documents in the batch) and `index` (0-based position) alongside the existing metadata/binary — useful for loops and for telling the empty result apart via `{{ $json.success }}`.

#### 1.13.0 - 2026-06-16

- Document Sync: added "Check & Download New Documents" operation — combines "Check New Documents" and "Download Document File" into one step. It claims a batch from the sync queue and, using the batch-wide lease token, downloads the file for each `document.created` entry, returning one item per document with the metadata in `json` and the file in `binary`. Non-file events (e.g. `document.deleted`) pass through as metadata only; a single failed download (file vanished/skipped server-side between claim and download) is recorded on that item (`downloadError`/`httpCode`) without aborting the rest of the batch. The lease token is included in the output so a downstream "Report Transfer Status" can still acknowledge each entry. The low-level "Check New Documents" and "Download Document File" operations remain available.
